## 🧩 选型背景与核心思路

在小红书（RedNote）的数据采集场景中，核心难点通常在于**风控对抗**（如 IP 封禁、Cookie 签名校验）。本次选型主要对比了三种主流路径：**客户端去中心化采集（插件）**、**SaaS 服务（API）** 以及 **服务端集中采集（自建）**。

我们的目标是寻找一种**低成本、高安全性且可持续**的解决方案。

## 🔍 详细方案解析

### 1. 浏览器插件模式（最终推荐 🏆）
这是最“讨巧”的方案。它不试图从服务器端去攻破小红书的防线，而是**利用用户的真实身份**。
*   **优势逻辑**：
    *   **天然过风控**：请求直接从用户的真实 IP 和浏览器发出，携带真实的 Cookie 和指纹，极难被判定为机器人。
    *   **分布式架构**：相当于把服务器的压力分散到了每个用户的电脑上。即使某个用户的账号被限制，也不会导致整个系统瘫痪。
*   **实施建议**：基于开源项目 [social-media-copilot](https://github.com/iszhouhua/social-media-copilot) 进行二次开发，定制 UI 和数据导出逻辑即可。

### 2. 第三方 API 模式
*   **本质**：花钱买时间。你不需要关心 IP 和算法，只需要关心钱。
*   **风险点**：这属于灰色产业链。一旦厂家被起诉或跑路（Rug Pull），你的业务会瞬间停摆。且随着小红书风控升级，API 的单价通常会越来越贵。

### 3. 自建服务端模式 (MediaCrawler)
*   **本质**：自己造轮子。
*   **痛点**：虽然 [MediaCrawler](https://github.com/NanmiCoder/MediaCrawler) 封装了很好的逻辑，但**IP 池是最大的拦路虎**。小红书对机房 IP 的封锁非常严格，你需要购买昂贵的住宅代理 IP 才能保证稳定运行，维护成本远超预期。

---

## 💡 最终决策与建议

综合考虑**合规性、成本控制与稳定性**，我们决定采用 **方案 1：浏览器插件**。

**决策依据：**
1.  **降维打击**：利用“用户浏览器”这一天然白名单环境，避开了最耗费精力的 IP 对抗环节。
2.  **成本最优**：无需采购代理 IP，无需昂贵的服务器集群。
3.  **可扩展性**：直接基于成熟开源项目修改，开发周期短，见效快。
